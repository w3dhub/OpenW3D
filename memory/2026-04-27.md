## Summary: Why This Approach Is Hitting a Wall

**Problem**: ww3d2's rendering layer is deeply coupled to D3D9 interfaces. Stubbing incomplete types (forward declarations only) doesn't provide the method implementations that actual D3D code needs.

**Disabled files so far**:
- ddsfile.cpp - D3D texture loading
- mapper.cpp, matrixmapper.cpp - texture coordinate transforms
- shader.cpp - D3D shader state  
- surfaceclass.cpp - D3D surface operations
- texture.cpp - D3D texture management
- textureloader.cpp - D3D texture loading from files
- texturethumbnail.cpp - texture preview generation

**Still failing**: vertmaterial.cpp and many others (78 errors)

**The fundamental issue**: Even with all the stub definitions, we can't compile because:
1. Incomplete struct types (`struct _D3DMATERIAL9`) don't have members accessible
2. COM interface methods require actual implementation, not just stubs
3. The rendering pipeline is: ww3d2 → DX8Wrapper → D3D9 interfaces

**What actually works**: wwlib compiles cleanly. The bgfx backend itself would compile if we can get the DX8 abstraction layer working.

**Alternative approach needed**: Instead of stubbing individual files, we need to either:
1. Make DX8Wrapper an abstract interface with bgfx implementation
2. Or provide a complete minimal D3D9 stub library


## Key Decision Point

The current "disable D3D files" approach is working for some files but hitting fundamental limits. **The real solution** is to complete the bgfx backend that was started in PR #116:

1. The bgfx backend files exist (or should exist) in `Code/ww3d2/backends/`
2. They need to implement the DX8 rendering interface with bgfx calls
3. The D3D-dependent code should be completely excluded via CMake when `WANT_DX9=OFF`

**New plan**: Instead of trying to stub D3D interfaces one-by-one, focus on completing the bgfx backend implementation. The DX8Wrapper should route to bgfx when `ENABLE_BGFX_BACKEND=ON`.
